# mcp-sdks-programmatic-clients

Automatic **progressive tool discovery** and **programmatic tool calling**
for MCP client SDKs, implemented once in Rust and bound into each SDK.
Based on the patterns in
[MCP Client Best Practices](https://modelcontextprotocol.io/docs/develop/clients/client-best-practices.md)
and the production implementations in [goose](https://github.com/block/goose).

This repo is a working proof-of-concept with an end-to-end TypeScript demo.
See [`PLAN.md`](./PLAN.md) for the design rationale.

Other language bindings (Python/Go/Swift/Kotlin/C#/Ruby) are designed but
not built; see `PLAN.md` §4.2.

## The one-line integration

```ts
import { Client } from '@modelcontextprotocol/client';
import { augment } from '@modelcontextprotocol/client-augmentation';

const client = new Client({ name: 'demo', version: '0.0.1' });
await client.connect(transport);

const agent = augment(client, {
  progressiveDiscovery: { contextPctThreshold: 0.03 },
  codeMode: { enabled: true },
});

// Same surface as before — but automatic.
const { tools } = await agent.listTools();
const result = await agent.callTool({ name: 'search_tools', arguments: { query: 'logs' } });
```

Remove the `augment(...)` line and host code is back to the stock SDK.

## How to run

### 1. Typecheck and test the Rust core

```sh
cargo check --workspace
cargo test -p mcp-client-core
```

Expected: clean; 2 tests pass (BM25 ranking + cache invalidation).

### 2. Build the Node addon

```sh
cd crates/mcp-client-node
pnpm install
pnpm run build:debug    # ~30s; `build` does a release build, ~60s
```

Produces `mcp-client-node.<platform>-<arch>.node` plus auto-generated
`index.js` / `index.d.ts`.

### 3. Run the TypeScript demo

```sh
cd typescript-sdk
pnpm install --filter '@modelcontextprotocol/examples-augmentation-demo...'
pnpm --filter '@modelcontextprotocol/examples-augmentation-demo' typecheck
pnpm --filter '@modelcontextprotocol/examples-augmentation-demo' demo
```

### 4. Enable code-mode (optional)

Code-mode requires the `code-mode` Cargo feature. First build costs
~5–10 minutes because `pctx_code_mode` transitively pulls in `deno_core`
and `v8`:

```sh
cargo check -p mcp-client-core --features code-mode
cargo test  -p mcp-client-core --features code-mode
```

Then rebuild `mcp-client-node` pointing at that feature if you want the
Deno-backed `execute_typescript` tool exposed through the JS binding.
(Currently the node crate doesn't propagate the feature; that's a
one-line Cargo.toml change to do in phase 2.)

## Design highlights

- **One source of truth.** BM25 scoring, cache invalidation, prompt-cache
  preservation, `list_changed` semantics, and code-mode sandbox plumbing
  all live in `mcp-client-core`. Language bindings are pure glue.
- **Core never opens sockets.** Each binding adapts the SDK's own
  `Client.listTools()` / `Client.callTool()` into an `McpConnection` via
  two callbacks. All auth, retries, OAuth, streaming — they stay inside
  the SDK that already does them well.
- **Minimal FFI contract.** Rust ↔ JS is two functions, each
  `(payload: string) => Promise<string>`. No complex objects cross the
  boundary. Makes every binding (Python/Go/…) drop-in once the core's
  shape is agreed.
- **Opt-in and additive.** `augment(client, …)` returns the same
  `listTools` / `callTool` shape. Removing the line restores stock
  behavior.
- **Production-lifted.** The catalog design mirrors goose's
  [`extension_manager.rs`](https://github.com/block/goose/blob/main/crates/goose/src/agents/extension_manager.rs);
  code-mode delegates to the same
  [`pctx_code_mode`](https://github.com/portofcontext/pctx) crate that
  goose's [`code_execution.rs`](https://github.com/block/goose/blob/main/crates/goose/src/agents/platform_extensions/code_execution.rs)
  uses in production.

## Next steps

See [`PLAN.md`](./PLAN.md) §5. In priority order:

1. **Ship prebuilt binaries** on npm (`aarch64-apple-darwin`,
   `x86_64-apple-darwin`, `x86_64-unknown-linux-gnu`,
   `aarch64-unknown-linux-gnu`, `x86_64-pc-windows-msvc`). `@napi-rs/cli`
   handles the CI matrix well.
2. **Python binding** via PyO3 / maturin. Proves the "bind once, use
   everywhere" thesis.
3. **Embedding-based retrieval** as an alternative `DiscoveryStrategy`,
   keeping BM25 as the default.
4. **Dynamic-server meta-tools** (`enable_server` / `disable_server`)
   with an SDK-side lifecycle hook.
